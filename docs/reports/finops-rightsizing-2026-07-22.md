## 1. Node pool — the only material cost lever ($131.36/mo, 72% of spend)

**The data — spend is high, utilization is very low:**
- Single resource `aks-system-…-vmss` = **$131.36 MTD**, i.e. essentially *all* compute spend.
- Pool = **2 × Standard_D2s_v7** (confirmed: 2 nodes × 2 idle-CPU series each = **4 vCPU total**, ~8 GB RAM/node).
- **Actual CPU utilization: ~9.1% and ~9.3%** per node right now; **7-day peak (`max_over_time`) only 8.1% / 9.3%** — i.e. it never meaningfully spikes. p95 == max == current, so this is a genuinely flat, idle-heavy profile, not an averaging artifact.
- **Memory: 38.4% / 27.5%** actual (~3–6 GB in use of ~16 GB).
- **The catch:** CPU *requests* are committed at **77.8%** of allocatable (memory requests 39.4%). This is why the autoscaler holds both nodes — the pods have *reserved* the CPU but are *using* ~9% of it. This is a textbook over-provisioned-requests-on-non-burstable-hardware pattern.

You are paying non-burstable D-series prices for hardware that runs at <10% CPU.

### Recommendation (a) — move to a burstable B-series SKU *(primary, highest ROI)*
Swap the node pool VM SKU from **Standard_D2s_v7 → Standard_B2ms** (same 2 vCPU / 8 GB, so memory headroom and the ~40% memory requests are preserved). B2ms has a **40% CPU baseline** and banks credits while idle — a workload sitting at 9% CPU accrues credits continuously and can still burst to 100% for the occasional gateway/agent spike.
- Per-node run-rate today is ~$85/mo (D2s_v7, eastus PAYG); B2ms is ~$60/mo.
- **Estimated saving: ~$25/node/mo × 2 nodes ≈ $50/mo (~30% of compute, ~17% of total bill)** while keeping **both nodes** on-demand.
- Do **not** consider B2s (4 GB) — actual memory use (up to ~6 GB) plus the 39% memory-request commitment would risk OOM/eviction. B2ms is the safe burstable match.

> **Guardrail respected:** this changes VM *SKU only*. The system node pool autoscaler minimum stays at **2**. I am not recommending — and will not recommend — dropping it toward 1, given the two documented node-count outages. Both nodes remain; they just run on cheaper burstable silicon.

### Recommendation (b) — Spot user pool for non-critical MCP/agent pods *(secondary)*
The MCP server pods (github, azure-devops, kubernetes, prometheus, sre-mcp, cost-mcp) and the A2A agent pods are stateless and eviction-tolerant. Add a **separate Spot-priced user node pool** (up to ~80–90% discount vs on-demand) and pin those pods to it via nodeSelector + toleration on the Spot taint.
- This relieves the ~78% CPU-request pressure on the system pool (the real reason it can't consolidate), potentially letting the burstable system pool run comfortably on its min=2 without the agent workloads competing for reserved CPU.
- **Must stay on the on-demand system pool:** the gateway, PostgreSQL, and Redis (stateful / customer-facing — Spot eviction would drop the public endpoint). Only the non-critical MCP/agent tier moves to Spot.
- This is complementary to (a): do (a) first (simple SKU change, guaranteed saving), evaluate (b) once you want to push further.

- **Actual MTD cost: $0.00.** ACR is currently immaterial to the bill, so there is no dollar-denominated saving to chase this month — flagging as **low priority** per that reality.
- **However, capacity-vs-tier is worth a plain callout:** the registry is provisioned as **Standard** tier (per IaC — `acr.bicep`). Standard's list price is ~$20/mo for 100 GB included storage; Basic is ~$5/mo for 10 GB. The $0 MTD figure means either billing hasn't materialized this period or it's within a credit — verify against a full billing month before assuming Standard is "free."
- **Is Standard capacity actually being used?** This project pushes only a handful of small custom images (sre-mcp, cost-mcp, and the github/azure-devops stdio wrappers) — almost certainly well under **Basic's 10 GB** quota. Image pulls use **managed-identity AcrPull**, which works identically on Basic. I see no evidence in the project's IaC/runbooks of Standard-only features in use (**webhooks, scope-maps, or tokens** — none are configured).
- **Verdict:** capacity appears paid-for-but-unused *in principle*, but since the measured cost is currently $0, a downgrade to Basic is a **cleanup/hygiene item, not a saving** — do it only when convenient. *(Caveat: I have no direct ACR-storage query tool in this read-only set, so the "under 10 GB" claim is inferred from the known image inventory, not measured — confirm with `az acr show-usage` before downgrading.)*

## 3. Log Analytics & Key Vault — evaluated, mostly no action

### Log Analytics — **$45.02/mo (2nd-largest line item), but correctly priced → no rightsizing action**
This is the one place the report could look like blanket cost-cutting, so to be precise: $45/mo is real, but it is **consumption-based ingestion** on the **PerGB2018** tier (`log-contextforge-dev`) with **30-day retention** (retention ≤31 days is free). There is **no over-provisioned committed/capacity tier being wasted** — you pay only for GB ingested by Container Insights. From a *rightsizing/tier* standpoint (this report's mandate), **no action is recommended** — the resource is on the correct pay-as-you-go SKU with no idle committed capacity.
- *Informational only (not a rightsizing lever):* if you later want to reduce the $45 itself, that's a **data-volume** exercise — tune the Container Insights Data Collection Rule (drop verbose `cAdvisor`/stdout streams, raise the collection interval). That reduces ingested GB; it is not a mispriced-tier problem.

### Key Vault — **$0.17/mo, correctly sized → no action**
`kv-contextforge-dev` on Standard tier with RBAC auth, $0.17 MTD — this is per-operation transaction cost, exactly as expected. Standard (vs Premium/HSM) is the right tier for software-protected secrets. **No action recommended.**

### Virtual Network — **$5.20/mo, out of rightsizing scope**
Two static public IPs at $2.60 each (nginx ingress LB + AKS system LB). Both are architecturally required (documented in the Phase 3 SNAT fix). Not a rightsizing lever — **no action.**
